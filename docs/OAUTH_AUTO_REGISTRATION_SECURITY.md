# OAuth Auto-Registration Security

## Overview

DocuMCP intentionally allows **auto-registration of OAuth clients** without requiring authentication. This design decision balances developer convenience with security through comprehensive rate limiting and monitoring.

## Configuration

```env
# .env
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=false
```

**Why disabled?**
- Enables MCP clients (like `mcp-remote`) to self-register automatically
- Reduces friction for developers in private/trusted network deployments
- Acceptable risk when combined with strict rate limiting

## Security Controls

### Rate Limiting Protection

Rate limiting prevents abuse even with auto-registration enabled:

```php
// app/Providers/AppServiceProvider.php
RateLimiter::for('oauth-register', function (Request $request) {
    return [
        Limit::perHour(10)->by($request->ip()),
        Limit::perDay(50)->by($request->ip()),
    ];
});
```

**Limits**:
- **10 registrations per hour** per IP address
- **50 registrations per day** per IP address

These limits allow legitimate development workflows while preventing automated abuse.

### Additional OAuth Rate Limits

| Endpoint | Limit | Purpose |
|----------|-------|---------|
| Token (`/oauth/token`) | 30/min, 100/hour | Prevent brute force |
| Authorization (`/oauth/authorize`) | 30/min | Prevent abuse |
| Device Authorization | 30/min | CLI tool support |
| Device Verification | 5/min, 30/hour | Prevent user code enumeration |

### Additional Protections

1. **Registration Logging** - All registration attempts logged with:
   - Client ID
   - Client name
   - Software ID (if provided)

2. **Client Secret Rotation** - Secrets can be regenerated by admins

3. **Client Deactivation** - Admins can disable suspicious clients

4. **Bcrypt Hashing** - Client secrets stored with bcrypt, which is designed for password storage (unlike SHA-256)

## When to Require Authentication

Consider setting `OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=true` in these scenarios:

### 1. Public-Facing Deployment
**Risk**: Untrusted users on public internet
```env
# Production (public internet)
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=true
```

**Impact**:
- Admin approval required for all client registrations
- Eliminates automated abuse vectors
- Adds operational overhead

### 2. Abuse Detected
**Signs of abuse**:
- Same IP registering multiple clients rapidly
- Client names with spam/phishing indicators
- Unusual redirect URIs (non-HTTPS, suspicious domains)
- Hitting rate limits frequently

**Action**: Enable authentication and audit existing clients

### 3. Strict Compliance Requirements
**Scenarios**:
- Organization policy requires manual review of all integrations
- Regulatory compliance (HIPAA, SOC 2, etc.)
- High-security environments

### 4. Production Scaling Issues
**Indicators**:
- High-volume traffic makes manual client vetting necessary
- Database growth from unused/test clients
- Support burden from misconfigured clients

## Monitoring Registration Patterns

### Check Recent Registrations

```bash
# View recent OAuth client registrations
grep "OAuth client registered via RFC 7591" storage/logs/laravel.log | tail -20
```

### Query Database for Registration Patterns

```sql
-- Count registrations by day
SELECT DATE(created_at) as date, COUNT(*) as count
FROM oauth_clients
GROUP BY DATE(created_at)
ORDER BY date DESC
LIMIT 30;
```

**Red flags**:
- Unusually high registration counts in short timeframe
- Sequential client names (test1, test2, test3...)
- Non-descriptive names ("client", "app", "test")

### Monitor Client Activity

```sql
-- Find inactive clients (never issued tokens)
SELECT id, client_name, redirect_uris, created_at
FROM oauth_clients
WHERE id NOT IN (
    SELECT DISTINCT client_id FROM oauth_access_tokens
)
AND created_at < NOW() - INTERVAL '7 days';
```

### Alert on Rate Limit Violations

```bash
# Check for rate limit hits
grep "Too Many Attempts" storage/logs/laravel.log | \
  grep "oauth/register" | tail -20
```

If you see frequent rate limit violations, it indicates potential abuse.

## Abuse Response Procedures

### 1. Identify Malicious Client

```bash
# Find recently created clients
docker exec documcp_app php artisan tinker
>>> \App\Models\OAuthClient::where('created_at', '>', now()->subHours(24))->get();
```

### 2. Disable Client

```bash
# Deactivate client
docker exec documcp_app php artisan tinker
>>> $client = \App\Models\OAuthClient::find('client-id-here');
>>> $client->update(['is_active' => false]);
```

### 3. Revoke All Client Tokens

```bash
# Revoke access tokens
docker exec documcp_app php artisan tinker
>>> \App\Models\OAuthAccessToken::where('client_id', 'client-id-here')->delete();
>>> \App\Models\OAuthRefreshToken::where('client_id', 'client-id-here')->delete();
```

### 4. Block IP (if needed)

```bash
# Add to firewall (example - Ubuntu UFW)
sudo ufw deny from <malicious-ip> to any
```

### 5. Enable Authentication Requirement

```env
# .env - Require admin approval going forward
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=true
```

```bash
# Restart application
docker-compose restart app queue
```

## Best Practices

### For Private Deployments (Recommended: Auto-Registration Enabled)

✅ **Keep auto-registration enabled** if:
- Deployed on private network/VPN
- Behind firewall with IP whitelisting
- Used by trusted developers only
- Rate limiting is sufficient protection

✅ **Monitor logs weekly** for:
- Unusual registration patterns
- Rate limit violations
- Suspicious client names/URIs

✅ **Periodically audit clients**:
```bash
# List all clients
docker exec documcp_app php artisan tinker
>>> \App\Models\OAuthClient::all(['id', 'client_name', 'redirect_uris', 'created_at']);
```

### For Public Deployments (Recommended: Authentication Required)

✅ **Require authentication**:
```env
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=true
```

✅ **Implement admin approval workflow**:
- Review redirect URIs for legitimacy
- Verify client purpose before approval
- Document approved clients

✅ **Use additional IP filtering**:
- Traefik rate limiting (100 req/sec limit)
- IP whitelisting for known clients
- Cloudflare or WAF for DDoS protection

## Comparison: Auto-Registration vs Authentication Required

| Aspect | Auto-Registration Enabled | Authentication Required |
|--------|---------------------------|------------------------|
| **Developer UX** | No manual setup required | Requires admin approval |
| **Security** | Rate-limited | Manual review |
| **Abuse Risk** | Low (with monitoring) | Minimal |
| **Operational Burden** | Low | Higher (manual approvals) |
| **Best For** | Private deployments, dev teams | Public APIs, high-security environments |

## Recommended Configuration by Environment

### Development
```env
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=false
```
**Rationale**: Developer convenience, minimal risk

### Staging/Testing
```env
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=false
```
**Rationale**: Test auto-registration flow, monitor for issues

### Production (Private Network)
```env
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=false
```
**Rationale**: Trusted users, rate limiting sufficient, monitor logs weekly

### Production (Public Internet)
```env
OAUTH_CLIENT_REGISTRATION_AUTH_REQUIRED=true
```
**Rationale**: Untrusted users, manual approval safer, compliance requirements

## Security Checklist

Before going live with auto-registration enabled:

- [ ] Rate limiting configured (default: 10/hour, 50/day)
- [ ] Logging enabled and monitored
- [ ] Network secured (firewall, VPN, or Traefik rate limiting)
- [ ] Monitoring alerts configured for suspicious activity
- [ ] Abuse response procedure documented and tested
- [ ] Team trained on reviewing logs and disabling clients
- [ ] Periodic audits scheduled (weekly or monthly)
- [ ] Consider Apprise alerts for rate limit violations
- [ ] PKCE enforcement enabled for public clients (default: S256 required)
- [ ] Device Authorization Grant configured for CLI tools

## Conclusion

**Auto-registration works when combined with:**
1. Rate limiting (default: 10/hr, 50/day)
2. Logging
3. Regular monitoring
4. Documented abuse response procedures

**For DocuMCP's typical use case** (private deployment, trusted developers), auto-registration enabled is the recommended configuration. Public-facing deployments should require authentication.

**Questions to ask:**
- Are all users trusted?
- Is the deployment on a private network?
- Can you monitor logs regularly?
- Do you have procedures to respond to abuse?

If you answered "yes" to all four, auto-registration is appropriate. Otherwise, require authentication.
